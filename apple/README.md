# Anvil — Apple clients (macOS + iOS/iPadOS)

Hybrid shell: a SwiftUI app hosting the Anvil web client in a `WKWebView`, same as the Android
app. **The web UI is bundled in the app** (`Sources/Anvil/web`, generated by
`anvild/web/bundle-native.ts`) and served from a local custom scheme (`anvil-app://`), so the
shell + fonts load offline; only the daemon data connection (WS/REST over Tailscale) needs the
network — injected as `window.ANVIL_DAEMON_URL`.

The Swift sources are shared across platforms: `WebView` conforms to `NSViewRepresentable` on
macOS and `UIViewRepresentable` on iOS (`#if os(...)`); the macOS-only menu commands / `Updater`
and the iOS-only push layer (`Push.swift`) are each guarded. One universal iOS binary covers
iPhone, iPad, and iPad Mini (`TARGETED_DEVICE_FAMILY = 1,2`).

## Build & run (no Xcode.app needed)

```sh
# 1. (re)generate the bundled web assets — run after any web client change:
cd anvild && bun run build:web && bun run web/bundle-native.ts ../apple/Sources/Anvil/web

# 2. compile / run the macOS app:
cd apple
swift build      # compiles
swift run        # launches the window
```

`swift run` runs the SPM executable directly (uses `Bundle.module` for the bundled web). This is
the canonical local build with the Command Line Tools — **no Xcode.app required**.

## Producing a distributable .app

```sh
cd apple
./make-app.sh            # → apple/dist/Anvil.app  (bundles web, builds release, signs)
./make-app.sh --skip-web # reuse the existing Sources/Anvil/web bundle (faster)
./make-app.sh --debug    # package the debug build
open dist/Anvil.app
```

`make-app.sh` needs only the Command Line Tools (`swift`, `iconutil`, `codesign`) — **no
Xcode.app**. It lays out `Anvil.app/Contents/{MacOS,Resources}`, copies the bundled web client to
`Contents/Resources/web` (served at runtime via `Bundle.main`), compiles the `.appiconset` into
`AppIcon.icns`, writes `Info.plist` (version pulled from `project.yml`), and **ad-hoc** codesigns
the bundle. Ad-hoc signing means no Gatekeeper/notarization — fine for local/Tailscale-internal
distribution; recipients may need to right-click → Open the first time. Notarized distribution and
iOS (device install, push, TestFlight) require full **Xcode.app** + the Apple Developer Program.

## Configure the daemon URL

Defaults to the Tailscale URL. Override:

```sh
defaults write com.gte619n.anvil anvil.baseURL "https://your-host.ts.net:7701/"
```

## iOS / iPadOS (needs Xcode + Apple Developer Program)

The iOS app is a second xcodegen target (`Anvil-iOS` in `project.yml`) sharing the same Swift
sources. Unlike the macOS shell, it **requires full Xcode.app** (SwiftPM can't build an iOS app)
and — for push / device install / TestFlight — the **Apple Developer Program** ($99/yr).

```sh
cd apple
# 1. (re)generate the bundled web assets (same as macOS):
( cd ../anvild && bun run build:web && bun run web/bundle-native.ts ../apple/Sources/Anvil/web )
# 2. generate the Xcode project and open it:
xcodegen generate        # brew install xcodegen
open Anvil.xcodeproj
# 3. select the "Anvil-iOS" scheme + a Simulator or device, set your signing team, Run.
```

Push setup (see `docs/plans/anvil-ios-ipados-clients.md`): register App ID `com.gte619n.anvil`
with the Push Notifications capability, create a token-based APNs key (`.p8`), and drop it at
`~/.config/anvil/apns-key.json` (`{ keyId, teamId, bundleId, key, production }`) on the daemon
host. Flip `aps-environment` in `Resources/Anvil-iOS.entitlements` (and `production` in the key
file) to `production` for TestFlight builds.
